Overview

The Virtru Common Operating Picture (COP) is a full-stack geospatial situational awareness application that demonstrates data-centric security using the Virtru Data Security Platform (DSP). All sensitive object data is encrypted at rest using the Trusted Data Format (TDF) while retaining plaintext metadata for map visualization and search.

COP ingests structured data objects (vehicles, sensors, personnel, etc.) through a schema-driven UI, encrypts payloads as NanoTDF blobs, stores them in a PostGIS-enabled PostgreSQL database, and streams real-time updates to connected clients via gRPC. Authorized users can decrypt and view protected content directly in the browser using Web Workers.

Startup Flow

How COP boots from go run . serve to serving requests:

1
Entry point — main.go calls cmd.Execute() and embeds frontend assets via embed.go
2
Command dispatch — cmd/startCmd.go handles the serve Cobra command
3
Config loading — pkg/config/config.go loads config.yaml via Viper; resolves DB URL, platform endpoint, OIDC settings, TLS, ports
4
Server init — api/server.go creates CopServer with DSP client, database pool, and entitlements cache (Ristretto)
5
gRPC server start — Connect RPC server registers TdfObjectServiceHandler and TdfNoteServiceHandler on port :5002 with h2c (HTTP/2 cleartext)
6
Static server start — HTTP server on :5001 serves the embedded Vite build with environment variable injection (pkg/ui/)
7
PG listener start — api/pglistener.go subscribes to tdf_objects_inserted and tdf_note_object_inserted PostgreSQL channels; relays inserts to streaming clients

▶api/tdfobjectserver.go — RPC HandlersCore

Implements all Connect RPC service interfaces. Key handlers:

  • CreateTdfObject — validates, stores encrypted TDF blob in tdf_objects table
  • QueryTdfObjects — executes filtered DB query (geo, time range, JSONB, source type)
  • StreamTdfObjects — opens server-sent stream; pglistener delivers new inserts in real-time
  • GetEntitlements — fetches user entitlements from DSP, caches per user token (15-min TTL)
  • ListSrcTypes / GetSrcType — returns source type schema definitions from DB
▶api/pglistener.go — Real-time RelayCore

Subscribes to PostgreSQL NOTIFY channels and relays row data to active streaming clients:

channels:
  - "tdf_objects_inserted"      // fires on INSERT to tdf_objects
  - "tdf_note_object_inserted"  // fires on INSERT to tdf_notes

flow: PG NOTIFY → pgxlisten → channel broadcast → gRPC stream → client
▶pkg/dspClient & pkg/tdfPackages

Wrappers around the OpenTDF platform SDK:

  • pkg/dspClient/ — initializes sdk.SDK with platform endpoint, OIDC client credentials, KAS URL
  • pkg/tdf/ — EncryptBytes() and DecryptBytes() using NanoTDF format

▶src_types.metadata JSONB StructureSchema

The metadata column drives the UI configuration for each source type:

{
  "geoField":     "location",          // which form field maps to PostGIS geometry
  "searchFields": ["callsign","unit"], // fields indexed in the search JSONB
  "attrFields":   ["clearance"],       // TDF attribute bindings
  "displayConfig": { ... },            // UI display rules
  "mapConfig": {                       // map marker configuration
    "iconType": "milsymbol",
    "sidc": "SFGPUCI--------"          // MIL-STD-2525 Symbol ID Code
  }
}
▶PostgreSQL Triggers & NOTIFYReal-time

Each table has an AFTER INSERT trigger that publishes to a notification channel:

-- tdf_objects trigger
CREATE TRIGGER tdf_objects_inserted_trigger
AFTER INSERT ON tdf_objects
FOR EACH ROW EXECUTE FUNCTION notify_tdf_objects_inserted();

-- Function sends: NOTIFY tdf_objects_inserted, '<row JSON>'

The Go backend's pglistener receives these notifications and pushes them to all active gRPC streams.

▶Geospatial QueriesPostGIS

The ListTdfObjects query supports geographic bounding box filtering:

-- Filter by bounding box
WHERE ST_Within(geo, ST_MakeEnvelope($minLon, $minLat, $maxLon, $maxLat, 4326))

-- Filter by JSONB search fields
AND search @> $searchFilter::jsonb

-- Filter by source type
AND src_type = $srcType

Data Flows

Object Creation Flow

1
User fills form — React renders dynamic form from src_type.form_schema + ui_schema
2
Frontend encrypts — useTdf.tsx calls DSP SDK to encrypt form JSON as NanoTDF; resolves attribute bindings from src_type.metadata.attrFields
3
RPC call — CreateTdfObject sends TDF blob, plaintext search fields, geo coordinates, and metadata to Go server
4
DB insert — Go server writes row to tdf_objects; INSERT trigger fires NOTIFY tdf_objects_inserted
5
Stream relay — pglistener receives notification, marshals row, sends to all active StreamTdfObjects streams
6
Map update — Connected clients receive the new object; map marker appears immediately

Object Query & Decrypt Flow

1
User applies filters — SearchFilter.tsx collects source type, time range, geo bounds, JSONB predicates
2
QueryTdfObjects RPC — Server executes PostGIS + JSONB query; returns encrypted blobs with plaintext metadata
3
Worker pool decryption — useRpcClient.tsx distributes TDF blobs across 4 Web Workers; workers call DSP SDK decrypt
4
KAS authorization — Each worker calls DSP KAS with user token; KAS validates entitlements via AuthZ service; returns wrapped key if authorized
5
Results merge — Decrypted form data merged with plaintext metadata (live telemetry); rendered in TdfObjectResult.tsx

TDF Encryption

Data-Centric Security: Every object payload is encrypted as NanoTDF before leaving the browser. The Go server never sees plaintext object data — it only stores and relays encrypted blobs.

What Is Encrypted vs Plaintext

FieldStorageReason
tdf_blobEncrypted (NanoTDF)Full object payload — only accessible to authorized users
search (JSONB)PlaintextIndexed fields needed for server-side search filtering
metadata (JSONB)PlaintextLive telemetry (speed, heading) needed for map display without decryption
geo (GEOMETRY)PlaintextLocation required for PostGIS spatial queries and map rendering

Encryption Flow (Form Submit)

User submits form
  → useTdf.encrypt(formData, attributeFQNs)
    → @virtru/dsp-sdk NanoTDFClient.encryptString()
      → DSP KAS: wrap symmetric key with attribute policy
      → Returns NanoTDF binary blob
  → CreateTdfObject RPC { tdf_blob: blob, geo: ..., search: ..., metadata: ... }
NanoTDF vs TDF: COP uses NanoTDF format (compact, ~30 bytes overhead vs 1KB+ for standard TDF). Configured via config.yaml → ui.form_submit_nano_tdf: true.

Common Pitfalls

GEOS not found at build time: The Go binary requires libgeos for PostGIS geometry. Use the Dockerfile for production builds, or install libgeos-dev locally. CGO must be enabled: CGO_ENABLED=1.
Entitlements cache stale: Entitlements are cached for 15 minutes per user token. If policy changes don't take effect immediately, wait for cache expiry or restart the server.
Worker pool not decrypting: Web Workers require HTTPS or localhost. If serving over plain HTTP on a non-localhost domain, TDF decryption will fail silently due to browser security restrictions.
geo column null: The form field configured as geoField in src_type.metadata must resolve to a valid GeoJSON geometry. If the field is missing or malformed, geo stores as NULL and the object won't appear on the map.
